## Needs Human Review

### LOW-001: `DnD5eAdapter.applyHDRestore` reads/writes incorrect field paths

**Severity:** 🟠 HIGH (latent — currently unreachable via built-in content)

**Trigger scenario:** GM creates a custom cooked food item with `flags["ionrift-respite"].wellFed = true` and `flags["ionrift-respite"].buff = { type: "hit_die", formula: "1" }`. Player eats this food. `MealPhaseHandler._resolveBuff` → case "hit_die" → `_restoreHitDice(actor, 1)` → `adapter.applyHDRestore(actor, 1)` → silently does nothing → returns `amount` claiming success.

**Root cause:**
`DnD5eAdapter.applyHDRestore()` (line 172) reads from `cls.system?.hitDice?.spent` which does not exist in either dnd5e v3 or v4+. The correct paths are:
- v4+: `cls.system?.hd?.spent` (read) / `"system.hd.spent"` (write)
- v3: `cls.system?.hitDiceUsed` (read) / `"system.hitDiceUsed"` (write)

The parallel implementation in `RecoveryHandler._recoverHitDice` (line 382–429) correctly handles both versions. The adapter implementation has never worked, but was only surfaced as a live code path by commit `b605bdc` which routes `MealPhaseHandler._restoreHitDice` through the adapter (previously the adapter was bypassed with a direct fallback).

**Why not fixed:**
- No built-in content defines a `hit_die` buff type. The only buff types used in stub recipes and built-in items are `temp_hp`, `heal`, `advantage`, `resistance`, and `exhaustion_save`.
- The bug is only triggerable via user-created custom food items.
- The primary HD recovery path (`RecoveryHandler._recoverHitDice`, called during rest resolution) does NOT use the adapter and works correctly.
- Impact is "silent failure to gain a benefit" rather than data loss or crash.

**Recommended action:** Fix the field paths in `DnD5eAdapter.applyHDRestore` to mirror `RecoveryHandler._recoverHitDice`. Low urgency since no content triggers it today, but should be fixed before documenting `hit_die` as a supported buff type for content creators.

**Proposed fix:**
```javascript
async applyHDRestore(actor, count) {
    const classes = (actor.items ?? []).filter(i => i.type === "class");
    if (!classes.length) return;
    const useNewField = classes[0]?.system?.hd !== undefined;
    const sorted = [...classes]
        .map(cls => ({
            item: cls,
            spent: useNewField
                ? (cls.system?.hd?.spent ?? 0)
                : (cls.system?.hitDiceUsed ?? 0)
        }))
        .filter(c => c.spent > 0);
    let remaining = count;
    for (const cls of sorted) {
        if (remaining <= 0) break;
        const restore = Math.min(cls.spent, remaining);
        if (useNewField) {
            await cls.item.update({ "system.hd.spent": cls.spent - restore });
        } else {
            await cls.item.update({ "system.hitDiceUsed": cls.spent - restore });
        }
        remaining -= restore;
    }
}
```

## Observations

### Pre-existing: `RecoveryHandler._recoverHitDice` unawaited embedded document update

`actor.updateEmbeddedDocuments("Item", updates)` at line 425 is called without `await`. The HD write races with subsequent HP and exhaustion writes. In practice, Foundry batches updates per actor so this is unlikely to cause corruption, but it's technically unsafe. Not introduced in the last 24h — present since initial implementation.

### Pre-existing: `travelMishapRecovery` flag cleared during calculation

`RestFlowEngine._calculateRecovery` clears the `travelMishapRecovery` actor flag as a side effect (lines 287–288, 416–417). If `RecoveryHandler.applyAll` is skipped (e.g., `restRecoveryDetected` is true), the flag is consumed without benefit. Narrow interaction: requires both travel mishap AND rest-recovery module active.

### Design note: Watch roster append-only

`RestFlowEngine.registerChoice` appends to `watchRoster` without removing prior entries for the same character. However, investigation confirms this is safe: `registerChoice` is called exactly once per character at activity submission time. The engine is always fresh or deserialized with the correct roster state.

### Design note: Training streak reset scope

`ActivityResolver.resolve` (line 143) resets ALL diminishing-return flags when a non-training activity is chosen. This is intentional per commit message: "Spacing training across other camp activities resets the streak."

### Socket handler validation gaps (low risk)

Several campfire minigame socket handlers (`campCeremonyStageWood`, `CAMPFIRE_STRIKE`, etc.) lack input shape validation beyond truthy checks. All are gated by `game.user.isGM` on the receive side and operate within a trusted tabletop socket model. No data corruption risk in normal use.
